Module 1 — AD DS Fundamentals

The logical and physical anatomy of a directory: forests, trees, domains, OUs, sites, domain controllers, and the objects that live inside them.

After this module you can
  • Explain what AD DS does and which protocols it stands on (LDAP, Kerberos, DNS)
  • Distinguish forests, trees, domains, and OUs — and know which one is the security boundary
  • Describe the four directory partitions and the role of the global catalog
  • Choose correct group types and scopes, and apply the AGDLP nesting pattern
  • Explain how sites, subnets, and site links shape logon traffic and replication

1.1 · What AD DS actually is

Active Directory Domain Services (AD DS) is a distributed, replicated database of security principals — users, computers, groups, services — plus the authentication and authorization machinery around it. When a user signs in to a domain-joined PC, when a file server decides whether that user can open a share, when a printer is located, when a policy is applied to 5,000 laptops at once: that's AD DS doing its job.

In plain English — for every stakeholder

Think of Active Directory as a company's badge office and employee directory combined. It keeps the master list of every person, computer, and team; it checks IDs at every door (that's authentication); it decides which doors each badge opens (authorization); and it pushes company rules onto every workstation automatically (policy). When AD is down, nobody's badge works — which is why the rest of this course treats it as the most important system in the building.

It's useful to think of AD DS as three things fused together:

  1. A database. The directory itself, stored in NTDS.dit on every domain controller, queried over LDAP (TCP 389, or 636 for LDAPS). It's hierarchical, object-oriented, and optimized for reads.
  2. An authentication service. Kerberos v5 is the primary protocol: users get a Ticket Granting Ticket (TGT) at logon and present service tickets to resources, so passwords never travel to member servers. NTLM survives as a fallback for legacy scenarios.
  3. A policy engine. Group Policy (Module 3) rides on top of the directory to configure and secure every domain-joined machine centrally.
Non-negotiable dependency

AD DS cannot function without healthy DNS. Clients find domain controllers by querying DNS SRV records such as _ldap._tcp.dc._msdcs.corp.contoso.com. Roughly half of all "AD is broken" tickets are actually DNS problems. When troubleshooting anything in this course, check DNS first.

1.2 · The logical structure: forests, trees, domains

AD DS organizes itself in nested containers. From the outside in:

A forest is the top of everything — one or more domains sharing a common schema (the definition of what object types can exist), a common configuration partition, and a common global catalog. Every domain in a forest automatically trusts every other domain via transitive trusts. Because of that, the forest — not the domain — is the true security boundary. An attacker with Domain Admin rights in any domain of a forest can realistically escalate to the entire forest; this fact drives the ESAE design in Module 4.

A tree is a set of domains within a forest that share a contiguous DNS namespace: corp.contoso.com and its child emea.corp.contoso.com form a tree. A second tree in the same forest could be fabrikam.com — separate namespace, same schema, same forest.

A domain is the unit of replication for user/computer data, of authentication, and of administrative policy. Each domain has at least one domain controller, its own password and account policies (by default), and its own Domain Admins group.

In plain English — for every stakeholder

Analogy: the forest is the whole corporate group; each domain is one company inside it; a tree is a set of companies sharing a family name (contoso.com and emea.corp.contoso.com). Here's the counterintuitive part executives should hear: the locks between companies in the same group are mostly decorative. If a burglar fully takes over any one company, they can realistically reach them all — so the real fence line is the forest, and that fact drives every security decision in Module 4.

The four directory partitions

The NTDS.dit database on each DC isn't one blob; it's divided into naming contexts (partitions) with different replication scopes:

Directory partitions and replication scope
PartitionContainsReplicates to
SchemaDefinitions of every object class and attribute the forest can holdEvery DC in the forest
ConfigurationForest topology: sites, site links, services, the list of all domains and DCsEvery DC in the forest
DomainThe actual users, computers, groups, and OUs of one domainEvery DC in that domain only
ApplicationApp-specific data with a custom replication scope — DNS zones are the classic case (ForestDnsZones, DomainDnsZones)Only DCs you enroll (Module 4.5)

The global catalog

Because domain partitions don't cross domain lines, a multi-domain forest needs a way to search everything. A global catalog (GC) server holds a full copy of its own domain partition plus a partial, read-only replica (a curated subset of attributes, the Partial Attribute Set) of every other domain in the forest. GCs answer forest-wide searches on TCP 3268, resolve universal group memberships during logon, and translate user principal names (jsmith@contoso.com) to accounts. No reachable GC in a multi-domain forest generally means failed logons — memorize that for Module 2's design discussion.

Functional levels

Domain and forest functional levels gate which AD features are available, based on the oldest DC operating system you still allow. Raising the level (e.g., to Windows Server 2016 — the level required for features like Privileged Access Management) is one-way in practice, so you raise it only after every DC is upgraded. Windows Server 2025 introduced the first new functional level since 2016 (WinThreshold → Windows2025).

1.3 · Organizational units and delegation

An organizational unit (OU) is a container inside a domain. OUs exist for exactly two reasons, and designing them for anything else causes pain:

  1. Group Policy targeting. GPOs link to OUs (Module 3). Machines and users are placed in OUs so that policy lands on the right population.
  2. Delegation of administration. You grant the Milan helpdesk permission to reset passwords only on OU=Milan-Users via the Delegation of Control wizard — without making anyone a Domain Admin.

OUs are not security principals — you cannot put an OU on an ACL or grant it access to a share. That's what groups are for. A common beginner error is building OU trees that mirror the org chart ("VP of Marketing → Marketing → Social Media Team"). Org charts change quarterly; policy and delegation needs don't. Most mature designs use a shallow model split first by object type (Workstations / Servers / Users / Groups / Service Accounts), then by region or function only where policy or delegation actually differs.

Containers vs OUs

The default CN=Users and CN=Computers objects are containers, not OUs: you cannot link GPOs to them or delegate cleanly. Use redircmp and redirusr to redirect newly joined computers and new users into real OUs you control.

1.4 · Users, computers, and service identities

A user object is a security principal: it has a SID (security identifier) that never changes, a sAMAccountName (legacy logon name, CONTOSO\jsmith), a userPrincipalName (jsmith@corp.contoso.com), and a distinguishedName that encodes its location in the tree — CN=Jane Smith,OU=Users,OU=Chicago,DC=corp,DC=contoso,DC=com. ACLs store SIDs, not names, which is why renaming or moving a user never breaks their access.

Computer objects are security principals too — every domain-joined machine has an account and a password (rotated automatically every 30 days) and authenticates with Kerberos just like a person. This matters: computers can be members of groups, targets of policy, and subjects of attack.

For services, prefer group managed service accounts (gMSAs) over ordinary user accounts with never-expiring passwords: the domain rotates a gMSA's 240-character password automatically, and authorized hosts retrieve it transparently.

# Create a user, a computer object, and a gMSA with the ActiveDirectory module
New-ADUser -Name "Jane Smith" -GivenName Jane -Surname Smith `
  -SamAccountName jsmith -UserPrincipalName jsmith@corp.contoso.com `
  -Path "OU=Users,OU=Chicago,DC=corp,DC=contoso,DC=com" `
  -AccountPassword (Read-Host -AsSecureString "Initial password") -Enabled $true

New-ADComputer -Name FS01 -Path "OU=Servers,DC=corp,DC=contoso,DC=com"

# gMSA prerequisite (once per forest): create the KDS root key
Add-KdsRootKey -EffectiveImmediately
New-ADServiceAccount -Name svc-web01 -DNSHostName svc-web01.corp.contoso.com `
  -PrincipalsAllowedToRetrieveManagedPassword "WebServers"

1.5 · Groups: types, scopes, and the AGDLP pattern

Groups come in two types — security groups (have a SID, can appear on ACLs) and distribution groups (mail-only, no SID) — and three scopes that control who can be a member and where the group can be used:

Group scopes
ScopeCan containCan be granted accessTypical role
Global (G)Accounts and global groups from its own domainAnywhere in the forest (and trusting forests)Role/identity grouping: "EMEA-Finance-Staff"
Domain Local (DL)Accounts, global & universal groups from any trusted domainOnly within its own domainResource permissions: "FS01-FinShare-Modify"
Universal (U)Accounts, global & universal groups from any domain in the forestAnywhere in the forestCross-domain roles; membership stored in the GC

The canonical nesting pattern is AGDLP: Accounts go into Global groups (roles), global groups go into Domain Local groups (resource access), and Permissions are assigned to the domain local group. In multi-domain forests you insert Universal groups: AGUDLP. The payoff is that access reviews become readable ("who has Modify on the finance share?" — expand one DL group) and cross-domain changes never require touching ACLs.

In plain English — for every stakeholder

Two kinds of lists, two jobs. A global group is a job-title roster: "these people are the EMEA finance team." A domain local group is a door's access list: "these rosters may open the finance share." AGDLP just says: put people on rosters, put rosters on door lists, never staple an individual name directly to a door. When someone changes jobs you update one roster — every door fixes itself.

Token bloat

Every group in a user's transitive membership adds to their Kerberos token. Past roughly 1,000 group memberships, authentication starts failing in strange ways (HTTP 400s against IIS are a classic symptom). Deep, uncontrolled nesting is technical debt — audit it.

1.6 · The physical structure: sites, subnets, domain controllers

Everything so far was logical. AD's physical topology maps the directory onto real networks:

In plain English — for every stakeholder

A site is simply "a building" (or campus) in AD's mental map. Getting the map right matters for one homely reason: when you badge in each morning, the system should check your ID at the front desk in your own building — not phone a desk on another continent. Miss a building on the map, and that's exactly what happens: everything works, just mysteriously slowly.

A site is a set of well-connected IP subnets — in practice, "a location with LAN-speed links." You define subnet objects (e.g., 10.20.0.0/16 → BR-London) and AD uses them for two things that matter enormously:

  1. Client affinity. A logging-on client asks DNS for a DC, and the DC Locator process steers it to a domain controller in its own site. Miss a subnet definition and a laptop in London may authenticate against Chicago over the WAN — slow logons, slow GPO processing.
  2. Replication shaping. Within a site, DCs replicate changes within seconds over automatically generated connections. Between sites, replication is compressed, scheduled, and routed over site links whose cost and interval you control (default: every 180 minutes). Module 4 goes deep here.

A domain controller (DC) is a server holding a writable replica of the domain partition. AD DS is multi-master: you can change a user's phone number on any DC and replication converges the change everywhere — with a handful of single-master exceptions called FSMO roles (Module 2). A read-only domain controller (RODC) holds a non-writable replica and, by default, no password secrets, caching credentials only for accounts you explicitly allow. RODCs exist for branch offices and other locations where physical security of the server can't be guaranteed.

Knowledge check · Module 1

Prove the fundamentals

1. Which structure is the true security boundary in Active Directory?

The forest. All domains in a forest share a schema and transitive trust, and a Domain Admin in any domain can realistically escalate forest-wide. This is the founding assumption behind ESAE (Module 4).

2. A file server in corp.contoso.com needs a group to hold Modify permissions on a share, with members eventually coming from two domains. Per AGDLP, permissions should be assigned to a…

Domain local security group. DL groups can contain global/universal groups from any trusted domain and are used to hold the actual resource permission — Accounts → Global → (Universal) → Domain Local → Permission.

3. Users in a new branch office complain of very slow logons; you discover they authenticate against a DC on another continent. Most likely cause?

Missing subnet-to-site mapping. The DC Locator uses the client's subnet to find a same-site DC; unmapped subnets mean the client can be handed any DC in the domain.

4. Which partition replicates only to domain controllers you explicitly enroll?

Application partitions have administrator-defined replica sets — DNS's ForestDnsZones and DomainDnsZones are the everyday examples, and you'll build a custom one in Module 4.

2.1 · Deploying domain controllers

Since Windows Server 2012, dcpromo is gone; promotion is a two-step: install the role, then run one of three configuration cmdlets depending on what you're building.

# Step 1 — add the role binaries (all scenarios)
Install-WindowsFeature AD-Domain-Services -IncludeManagementTools

# Scenario A — first DC of a brand-new forest
Install-ADDSForest -DomainName "corp.contoso.com" -DomainNetbiosName "CORP" `
  -ForestMode WinThreshold -DomainMode WinThreshold `
  -InstallDns -SafeModeAdministratorPassword (Read-Host -AsSecureString)

# Scenario B — additional DC in an existing domain
Install-ADDSDomainController -DomainName "corp.contoso.com" `
  -SiteName "BR-London" -InstallDns -Credential (Get-Credential CORP\Administrator)

# Scenario C — new child domain in the forest
Install-ADDSDomain -NewDomainName "emea" -ParentDomainName "corp.contoso.com" `
  -DomainType ChildDomain -InstallDns -Credential (Get-Credential)

Three deployment practices separate professionals from tourists:

  • Server Core by default. A DC needs no GUI. Core cuts patch volume, reboot frequency, and attack surface. Manage remotely with RSAT, Windows Admin Center, and PowerShell remoting.
  • Install From Media (IFM) for slow links. ntdsutil "activate instance ntds" ifm "create sysvol full C:\IFM" on an existing DC produces media a new branch DC can seed from, so initial replication over a thin WAN ships only recent deltas.
  • RODC staged deployment. Pre-create the RODC account (Add-ADDSReadOnlyDomainControllerAccount) and delegate the attach to a branch admin who never receives domain-level rights. Define the Password Replication Policy deliberately: cache branch users, never cache admins.
The DSRM password

That SafeModeAdministratorPassword you set at promotion is the Directory Services Restore Mode password — a local account used when the DC boots with AD offline for restores. Vault it per-DC. Discovering during a disaster that nobody knows any DSRM password is a rite of passage you can skip.

2.2 · FSMO — the five single-master roles

Multi-master replication can't safely handle a few operations where two DCs acting simultaneously would corrupt the directory — schema changes, adding domains, issuing SID pools. AD assigns those to Flexible Single Master Operations (FSMO) role holders: two per forest, three per domain.

In plain English — for every stakeholder

AD lets every domain controller accept changes — like a company where any office can update the shared employee list. But five duties would cause chaos if two offices did them simultaneously (imagine two clerks handing out the same employee ID number, or two official wall clocks disagreeing). FSMO roles just say: for these five duties only, exactly one server is the designated notary. The table below is "which notary does what, and how you notice when he's on vacation."

The five FSMO roles
RoleScopeResponsibilityWhen it's down, you notice…
Schema MasterForestAll writes to the schema partitionOnly when extending the schema (Exchange, LAPS legacy, custom) — otherwise silent
Domain Naming MasterForestAdding/removing domains and application partitionsOnly during domain add/remove — otherwise silent
RID MasterDomainIssues blocks of Relative IDs (the unique tail of every new SID) to DCsSlowly: DCs exhaust their ~500-RID pools and object creation starts failing
PDC EmulatorDomainAuthoritative time source, immediate password-change target, lockout processing, GPO editing default, legacy compatFast and loud: time skew, "wrong password" oddities, lockout weirdness
Infrastructure MasterDomainUpdates cross-domain object references (phantoms)Stale cross-domain group members shown; irrelevant if every DC is a GC

The PDC Emulator is the role users feel. It sits at the top of the domain time hierarchy (configure it to sync from a reliable external source with w32tm), receives urgent replication of password changes, and is consulted by other DCs before rejecting an apparently-bad password — which is why a freshly changed password works everywhere instantly.

Transferring vs seizing

Transfer when the current holder is alive (graceful, coordinated). Seize only when the holder is dead and never returning — and a seized-from DC must never rejoin the network without being forcibly demoted and metadata-cleaned first, or you'll have two role holders fighting.

# Where are the roles right now?
Get-ADForest | Select-Object SchemaMaster, DomainNamingMaster
Get-ADDomain | Select-Object PDCEmulator, RIDMaster, InfrastructureMaster

# Graceful transfer of two roles to DC02
Move-ADDirectoryServerOperationMasterRole -Identity DC02 `
  -OperationMasterRole PDCEmulator, RIDMaster

# Seizure (dead holder only) — same cmdlet plus -Force
Move-ADDirectoryServerOperationMasterRole -Identity DC02 `
  -OperationMasterRole SchemaMaster -Force

2.3 · How many DCs, and where

DC placement is a set of trade-offs between logon locality, WAN cost, blast radius, and physical security. The working rules:

  • Two writable DCs per domain, minimum, always. One DC is not a design; it's a countdown. Put them on separate hypervisor hosts and, ideally, separate storage.
  • Per site, ask three questions: Can users tolerate WAN logons if the link drops? Is there local infrastructure (Exchange, file servers) that needs fast LDAP? Is the server physically secure? Yes/yes/yes → place a writable DC. Yes/yes/no → RODC. No local dependency and a reliable WAN → no DC at all; fewer DCs means fewer things to patch, monitor, and steal.
  • Global catalog placement: in a single-domain forest, make every DC a GC — there's no extra replication cost and it removes a failure mode. In multi-domain forests, put a GC in every site with users, or enable universal group membership caching for tiny sites where a full GC replica is too heavy.
  • Don't stack roles. DCs run AD DS and DNS. Not file services, not print, not the HR app. Every extra service is extra attack surface on the box that owns every password hash in the domain.
  • Virtualization is fine — with rules. Windows Server 2012+ hypervisors expose VM-GenerationID, which lets a DC detect snapshot rollback and quarantine itself (a new invocation ID, RID pool discard) instead of corrupting replication with a USN rollback. Never apply snapshots to DCs as an "undo" anyway; supported DC cloning exists for rapid scale-out.

2.4 · Backup and recovery

Directory data lives in NTDS.dit plus SYSVOL. A supported AD backup is a system state backup taken with a VSS-aware tool — Windows Server Backup is the built-in option:

Install-WindowsFeature Windows-Server-Backup
wbadmin start systemstatebackup -backupTarget:E: -quiet

Backups are only valid within the tombstone lifetime (180 days on modern forests). Restore an older backup and you reintroduce lingering objects — deleted objects the rest of the forest has already forgotten — which is its own outage (Module 4.4).

In plain English — for every stakeholder

Why restores are tricky here: every domain controller remembers the deletion. Restore a deleted file to one office and the other offices "helpfully" delete it again, because their records say it's gone. An authoritative restore stamps the recovered data "NEWEST — everyone take this copy," so it spreads back out instead of being re-shredded. The Recycle Bin below is the modern shortcut that skips the ceremony entirely.

The recovery decision tree

Choosing a recovery method
SituationMethodMechanics
One DC died; others healthyRebuild, don't restoreMetadata-cleanup the corpse (delete its object in AD Users & Computers / ntdsutil), promote a fresh replacement, let replication repopulate it
DC's database is corruptNon-authoritative restoreBoot into DSRM, restore system state, reboot; the DC then pulls everything newer from its partners — the restore is just a seed
Objects were deleted and the deletion has replicated everywhereAuthoritative restore or Recycle BinDSRM restore, then ntdsutil → authoritative restore → restore subtree "OU=Sales,DC=corp,DC=contoso,DC=com". This inflates version numbers by 100,000 so the restored objects win replication and propagate outward
Entire forest compromised/destroyedForest recoveryScripted, rehearsed rebuild from one root-domain backup: restore one DC per domain, seize roles, reset krbtgt twice, metadata-clean everything else, re-promote. If you haven't rehearsed it, you don't have it

The AD Recycle Bin — enable it today

With the Recycle Bin enabled (irreversible, requires 2008 R2+ forest level), deleted objects keep all attributes and links for the deleted-object lifetime and can be restored in seconds with no reboot, no DSRM, no ntdsutil:

Enable-ADOptionalFeature "Recycle Bin Feature" -Scope ForestOrConfigurationSet `
  -Target "corp.contoso.com"

# Restore an accidentally deleted user — group memberships intact
Get-ADObject -Filter 'samAccountName -eq "jsmith"' -IncludeDeletedObjects |
  Restore-ADObject

Complementing backups, ntdsutil snapshots (ntdsutil snapshot "activate instance ntds" create) capture a mountable point-in-time copy of the database you can expose with dsamain on an alternate LDAP port — perfect for answering "what did this group look like last Tuesday?" without restoring anything.

2.5 · Schema management

The schema defines every class (user, computer, group…) and attribute (mail, employeeID…) the forest can contain. Two properties make it dangerous: it's forest-wide (one copy, replicated to every DC), and additions are permanent — classes and attributes can be deactivated (defunct) but never deleted.

In plain English — for every stakeholder

The schema is the blank form that defines what fields an employee record can even have. Adding a field (say, "parking-spot number") changes the form for the entire corporate group, forever — you can retire a field, but never truly remove it. That's why schema changes get lab rehearsals, backups, and an empty admin group guarding the pen.

Schema changes happen more often than people expect: Exchange, SCCM/ConfigMgr, and adprep /forestprep during OS upgrades all extend it. The discipline for any extension:

  1. Rehearse in a lab forest restored from production media, and diff the schema before/after.
  2. Obtain proper OIDs for custom attributes from your enterprise OID arc — never invent numbers, and never reuse Microsoft's.
  3. Take a system state backup of the Schema Master first, run the extension on the Schema Master, and verify replication afterward with repadmin /showrepl.
  4. Guard membership of Schema Admins. Best practice is an empty Schema Admins group; add a named admin for the change window, remove them after. The Schema snap-in isn't even registered by default (regsvr32 schmmgmt.dll) — friction by design.
Indexing and the GC

When adding a custom attribute, decide whether it should be indexed (searched frequently?) and whether it belongs in the Partial Attribute Set (needed in forest-wide GC searches?). Both are flags on the attributeSchema object; both affect every DC's storage and replication.

Prove DC operations

1. Users report intermittent Kerberos failures and clocks across the domain have drifted several minutes apart. Which FSMO holder do you check first?

The PDC Emulator anchors the domain time hierarchy, and Kerberos tolerates only 5 minutes of skew by default. It's the FSMO role whose failure users feel fastest.

2. An OU with 200 users was deleted yesterday and the deletion has replicated to all DCs. The AD Recycle Bin was never enabled. Correct recovery?

A non-authoritative restore would be immediately overwritten by replication (the deletion is "newer"). Marking the subtree authoritative inflates its version numbers so the restored objects win and propagate. Enabling the Recycle Bin now can't recover objects deleted before enablement with attributes intact.

3. You seized the PDC Emulator role after DC01 suffered a motherboard failure. A week later a technician repairs DC01 and asks to power it back on in the datacenter. You should…

A DC whose roles were seized must never return as-is: it still believes it holds the role. Wipe or forcibly demote it offline and remove its metadata, then optionally re-promote it as a fresh DC.

4. Why does a snapshot-rollback on a pre-2012 hypervisor endanger a virtualized DC?

Rolling back rewinds the DC's Update Sequence Numbers. Partners' high-watermarks say they've already replicated those USNs, so new changes are never pulled — silent divergence. VM-GenerationID (2012+) lets the DC detect this and safely reset itself.

3.1 · What a GPO is — and where it physically lives

A Group Policy Object is a named bundle of configuration split into a Computer Configuration half (applies at boot/background refresh, per machine) and a User Configuration half (applies at logon, per user). "Configuration" spans registry policy, security settings, scripts, software deployment, folder redirection, and Group Policy Preferences.

In plain English — for every stakeholder

A GPO is a company handbook that enforces itself. Instead of an IT person walking to 5,000 laptops to set the screen-lock timeout, you write the rule once, attach it to the right department, and every machine applies it automatically — including ones bought next year. The rest of this module is really three questions: where do handbooks live, who receives which handbook, and what happens when two handbooks disagree.

Every domain GPO is stored in two places, and knowing this explains half of all GPO replication bugs:

  • The Group Policy Container (GPC) — an object in AD under CN=Policies,CN=System,DC=…, named by GUID, carrying version numbers and links. Replicates via AD replication.
  • The Group Policy Template (GPT) — a folder of the actual settings files in \\corp.contoso.com\SYSVOL\corp.contoso.com\Policies\{GUID}. Replicates via DFS Replication, a completely separate engine.

Two stores, two replication systems: when their version numbers disagree on a DC, clients get inconsistent policy. gpotool-style version checks and DFSR health are standard GPO troubleshooting steps.

Every domain ships with two default GPOs you should treat as protected infrastructure: the Default Domain Policy (linked at the domain — password, lockout, and Kerberos policy belong here and only here) and the Default Domain Controllers Policy (linked to the Domain Controllers OU — user rights on DCs). Best practice: never pile unrelated settings into them; create new GPOs, and keep dcgpofix in your back pocket to rebuild the defaults if they're ever mangled.

Local GPOs

Every Windows machine also has local GPOs (and per-user local GPOs since Vista). In a domain they're mostly irrelevant because they're processed first — meaning any domain policy overrides them, as the next section explains.

3.2 · Processing order and precedence: LSDOU

Group Policy applies in a strict order — Local → Site → Domain → OU → child OU ("LSDOU") — and the last writer wins. A setting configured in a GPO linked to the deepest OU beats the same setting from the domain, which beats site, which beats local. Settings that don't conflict simply merge.

In plain English — for every stakeholder

When handbooks disagree, the rule closest to you wins: a company-wide rule loses to a division rule, which loses to your own team's rule — sensible, because your team knows its needs best. The one exception: headquarters can mark a rule Enforced ("non-negotiable corporate law"), and then nothing below can override it. Every precedence puzzle in this module reduces to those two sentences.

Four modifiers bend the rules, and exam questions (and outages) live in their interactions:

  • Link order. Multiple GPOs on one OU are processed lowest-number-last… no — lowest link order (1) is processed last, so link order 1 has the highest precedence.
  • Block Inheritance (set on an OU) discards everything arriving from above. Use rarely; it makes behavior non-obvious.
  • Enforced (set on a link) makes that GPO immune to blocking and to being overridden below. Reserve for genuine "corporate law" — security baselines.
  • Security policy exception: account policies (password/lockout/Kerberos) for domain accounts are effective only when set at the domain level. A "stricter password policy" GPO on an OU silently does nothing to domain accounts — use Fine-Grained Password Policies (PSOs) for per-group differences instead.

Loopback processing

Normally user settings come from GPOs scoping the user's OU. On kiosks, RDS session hosts, and labs you want user settings determined by the machine. Loopback processing (Computer Config → Administrative Templates → System → Group Policy) does exactly that, in two flavors: Replace (ignore the user's own GPO list entirely) and Merge (apply the user's list, then re-apply the machine-scoped user settings on top).

When policy actually applies

Computers process policy at startup and users at logon (foreground), then everyone refreshes in the background every 90 minutes plus a random 0–30 minute offset (DCs: every 5 minutes). Some settings — software installation, folder redirection, drive mappings marked "run once" — apply only at foreground processing, which is why "just wait for refresh" sometimes never works and a reboot does. Security settings additionally re-apply every 16 hours even with no changes. Force a refresh with gpupdate /force or remotely with Invoke-GPUpdate.

3.3 · Scoping: links, security filtering, WMI filters, item-level targeting

Deciding who gets a GPO is a funnel with four gates, evaluated in order:

  1. Links. A GPO does nothing until linked to a site, domain, or OU. One GPO can link to many containers; a link can be disabled without deleting it.
  2. Security filtering. To apply, a user/computer needs both Read and Apply Group Policy permissions on the GPO. Default is Authenticated Users (which includes computers). Replace it with a specific group to narrow scope — but since the MS16-072 change, policy is read in the computer's context, so if you remove Authenticated Users from filtering you must still grant it (or Domain Computers) plain Read, or the GPO silently stops applying.
  3. WMI filters. A query evaluated at processing time — e.g., laptops only: SELECT * FROM Win32_SystemEnclosure WHERE ChassisTypes = 9 OR ChassisTypes = 10. Powerful, but every filter is a WMI query executed by every in-scope client at every refresh; keep them cheap and few.
  4. Item-level targeting (Preferences only) — per-setting conditions (group membership, OS, IP range, LDAP query) with a friendly UI. The most surgical tool in the box.
Design habit that saves careers

Many small, single-purpose GPOs with names like C-SEC-BitLocker-Laptops (scope prefix, category, purpose) beat one 900-setting monolith. Small GPOs diff cleanly, roll back cleanly, and can be delegated. And never edit a production GPO live during business hours — settings replicate and apply as you type; there is no "save" button.

3.4 · Administrative Templates, the Central Store, and Preferences

Administrative Templates are the registry-based bulk of Group Policy, defined by ADMX files (settings) plus language-specific ADML files (display text). By default each admin workstation reads its own local copies — so two admins on different Windows builds see different settings. The fix is the Central Store: copy an authoritative ADMX/ADML set to

\\corp.contoso.com\SYSVOL\corp.contoso.com\Policies\PolicyDefinitions\
\\corp.contoso.com\SYSVOL\corp.contoso.com\Policies\PolicyDefinitions\en-US\

Once that folder exists, every GPMC in the domain uses it automatically. Maintain it deliberately: refresh with each Windows release's templates, and add vendor templates (Chrome, Office, Adobe) here so every admin edits from the same catalog.

A true policy writes to the four Policies registry keys, is enforced (UI greys out), and reverts when the GPO no longer applies. Group Policy Preferences (drive maps, printers, scheduled tasks, registry, local groups, shortcuts…) instead tattoo ordinary registry/config once per refresh, users can change them afterward (unless you re-apply), and they offer item-level targeting. Rule of thumb: mandate with policies, provision with preferences.

Historical vulnerability worth knowing

GPP once allowed embedding credentials (e.g., local admin passwords) in XML stored in SYSVOL, "protected" by a published AES key — MS14-025 removed the UI, but attackers still hunt old Groups.xml files. Never store secrets in SYSVOL; for local admin passwords, deploy Windows LAPS.

4. A GPO edit made on DC02 shows on some clients but not others; GPMC on DC03 shows an older version of the GPO's settings files. The most likely failing component is…

The GPC (in AD) and GPT (in SYSVOL) replicate via different engines. Version mismatch between them — typically DFSR trouble — produces exactly this split-brain symptom.

4.1 · Trust relationships

A trust lets the security principals of one domain be authenticated and authorized in another. Three properties define every trust: direction (a one-way trust where A trusts B means B's users can access A's resources — access flows opposite to trust), transitivity (does the trust extend through to domains the partner trusts?), and how it's created (automatic vs manual).

In plain English — for every stakeholder

A trust is a badge-reciprocity agreement between two companies: "we'll honor each other's employee badges." Direction says whose badge works in whose building (and it's the reverse of the arrow — if we trust them, their people get into our building). Transitivity asks whether the deal extends to their partners too. The security levers further down are the fine print: check badges for forgery (SID filtering) and honor them only at specific doors (selective authentication).

The two security levers on every trust

SID filtering (a.k.a. quarantine) strips SIDs from an incoming user's token that don't belong to the trusted domain — defeating SID-history injection, where an attacker in the trusted forest forges a token claiming your Domain Admins SID. It's on by default for external and forest trusts; it gets disabled during migrations (SID history must flow) and then, dangerously, forgotten. Re-enable it when migration ends.

Selective authentication flips the trust from "everyone over there can authenticate here" to deny-by-default: trusted-forest users authenticate only to computers where you've granted them the Allowed to Authenticate permission. It's the right default posture for any trust with a partner you don't fully control.

4.2 · ESAE forests and the tiered administration model

Recall Module 1: the forest is the security boundary. The Enhanced Security Administrative Environment (ESAE) — the "red forest" — takes that literally: build a separate, minimal, hardened forest whose only inhabitants are the accounts and workstations used to administer production. A one-way trust is configured so production trusts the admin forest (admin-forest accounts can manage production), with selective authentication restricting where they can even log on. Compromise of production can't reach the admin accounts, because they don't live there.

In plain English — for every stakeholder

The idea in one sentence: don't store the master keys inside the building they open. ESAE builds a separate, tiny, heavily guarded building whose only purpose is housing the admin keys and the desks they're used from. Even a burglar who owns your entire main office finds no master keys inside — they were never there. The tier model below is the same logic applied internally: janitor keys, office keys, and vault keys never ride in the same pocket.

Underpinning ESAE — and outliving it — is the tier model: Tier 0 (anything that controls identity: DCs, AD CS, ADFS, backup of these), Tier 1 (servers/apps), Tier 2 (user workstations). The iron rule is credentials never flow down: a Tier-0 account must never log on to a Tier-1 or Tier-2 machine, because logging on leaves credential material where a lower-tier compromise can harvest it. Enforce with logon-right GPOs per tier, separate admin accounts per tier, and Privileged Access Workstations (PAWs) — hardened, internet-restricted machines that are the only place Tier-0 credentials are ever typed.

Current Microsoft guidance

Microsoft has retired ESAE as its default recommendation (it remains valid for isolated/regulated environments) in favor of the cloud-era enterprise access model and privileged-access strategy: PAWs, just-in-time elevation (PIM / PAM with the 2016 forest-level Privileged Access Management feature and time-bound group membership), and conditional access. The tier-separation principle is unchanged — only the machinery is. Expect both ESAE and its successor on modern exams and in real estates, since deployed red forests will run for years.

4.3 · Replication: how a change actually travels

Every DC keeps an Update Sequence Number (USN) counter it increments for each local write. Replication partners track two structures against it: the high-watermark ("the last USN I've seen from you, per partition") for the routine pull cycle, and the up-to-dateness vector ("the newest originating write I hold from every DC") which enables propagation dampening — the reason a change flooding through a mesh of DCs isn't replicated to the same DC five times. Conflicts (two admins edit the same attribute on different DCs inside one replication window) resolve by version number, then timestamp, then originating-DC GUID: last writer wins, no error raised.

In plain English — for every stakeholder

Replication is offices gossiping updates to the shared employee list. Each office keeps a change counter (the USN) and remembers "the last change number I heard from you" — so catching up means "tell me everything after #4,712," not re-reading the whole list. The up-to-dateness vector is the etiquette that stops the same piece of gossip being repeated to an office that already heard it from someone else.

Topology is generated for you: the Knowledge Consistency Checker (KCC) on each DC builds intrasite connection objects into a ring with extra edges such that no DC is more than three hops from any other; change notification makes intrasite convergence a matter of seconds. Between sites, the Inter-Site Topology Generator (ISTG) in each site elects bridgehead servers that exchange compressed traffic across site links you define, according to their cost (lower preferred) and schedule/interval (default 180 minutes, floor 15). All site links are transitively bridged by default; on non-fully-routed networks you disable bridging and build site link bridges by hand.

4.4 · Monitoring and troubleshooting replication

The diagnostic loop is short and repeatable:

# 1. Overall health — run both regularly, not just during incidents
repadmin /replsummary            # forest-wide failure/latency matrix
dcdiag /v /c /e /q               # comprehensive tests, errors only, every DC

# 2. Zoom into one DC's partners and last results
repadmin /showrepl DC03 /errorsonly

# 3. See what hasn't converged and why
repadmin /queue DC03             # backlog of pending inbound work
repadmin /showvector /latency "dc=corp,dc=contoso,dc=com"

# 4. Force the issue while watching
repadmin /syncall /AdeP          # all partitions, all partners, cross-site, push

# PowerShell equivalents for dashboards
Get-ADReplicationFailure -Scope Forest
Get-ADReplicationPartnerMetadata -Target DC03 | Select Partner, LastReplicationSuccess

Interpretation habits: a wall of failures to one DC usually means that DC (or the network path/firewall to it — AD needs RPC 135 + dynamic high ports, LDAP, Kerberos, SMB, DNS). Failures on one partition only often mean DNS or a GC issue. Error 8524 ("DSA couldn't be resolved via DNS") is a CNAME/SRV lookup failure for the partner's _msdcs GUID record — DNS again, always DNS.

Lingering objects and strict consistency

If a DC is offline longer than the tombstone lifetime and comes back, it holds lingering objects — things everyone else deleted and has since purged. With strict replication consistency (default on modern forests) partners refuse to replicate with it (event 1988/2042) rather than resurrect the dead. Remediate with repadmin /removelingeringobjects <bad-DC> <good-DC-GUID> <partition> — or, more honestly, demote and rebuild the offender. Prevention beats cure: no DC should ever be off (or snapshotted-away) for months.

4.5 · Custom application directory partitions

An application directory partition stores application data in AD with a replica set you choose — any set of DCs across any domains of the forest — and it never replicates to global catalogs. You use them when an app needs AD's replication, security, and LDAP access, but its data shouldn't burden every DC in a domain. You've been relying on them since Module 1: DNS's ForestDnsZones and DomainDnsZones are application partitions.

# Create a partition and manage replicas with ntdsutil (run as Enterprise Admin)
ntdsutil
  activate instance ntds
  partition management
  connections
    connect to server DC01.corp.contoso.com
    quit
  create nc dc=AppData,dc=corp,dc=contoso,dc=com DC01.corp.contoso.com
  add nc replica dc=AppData,dc=corp,dc=contoso,dc=com DC03.corp.contoso.com
  list                      # show all partitions and replica sets
  quit

Operational notes: creating and deleting partitions requires the Domain Naming Master to be online (Module 2 pays off); each partition designates a security descriptor reference domain that supplies default ACLs for its objects; deleting a partition destroys its data everywhere, so replicas first, regrets never. Query them like any naming context — Get-ADObject -SearchBase "dc=AppData,dc=corp,dc=contoso,dc=com" -Filter *.

Knowledge check · Module 4

Prove advanced administration

1. Users in partner forest fabrikam.com need access to exactly three servers in corp.contoso.com, and nothing else should even authenticate them. Best configuration?

Selective authentication turns the trust into deny-by-default: trusted users can authenticate only to computers where they hold Allowed to Authenticate. One-way (corp trusts fabrikam) since access flows opposite to trust direction.

2. In the tier model, why is a Tier-0 admin forbidden from logging on to a Tier-2 workstation — even "just quickly"?

Credential theft (pass-the-hash/ticket) harvests material left by interactive logons. The tier model's one rule — credentials never flow down — exists precisely to keep Tier-0 secrets off machines with Tier-2 exposure. GPOs can enforce it, but only if you configure them.

3. A change made on DC01 reaches DC02 and DC03, which both replicate with DC04 — yet DC04 receives it only once. The mechanism responsible is…

The up-to-dateness vector records the newest originating update DC04 holds from DC01; when DC03 offers the same originating write, DC04 declines it. The high-watermark handles the routine per-partner cursor, but cross-path duplicate suppression is the vector's job.

4. A branch DC powered off for seven months is reconnected. Partners log event 2042 and refuse to replicate with it. This behavior is…

Beyond the tombstone lifetime, the returning DC holds objects the forest has purged. Strict consistency quarantines it rather than resurrecting deletions. Remove lingering objects with repadmin — or demote and rebuild, which is usually cleaner.

5.1 · PKI foundations in five minutes

A digital certificate (X.509) binds a public key to an identity — a user, a computer, a service — under the signature of a certification authority (CA). Whoever holds the matching private key can prove they are that identity (authentication), receive data only they can read (encryption), and produce signatures only they could have made (integrity/non-repudiation). Certificates power smart card and certificate-based logon, LDAPS on your DCs, 802.1X Wi-Fi, HTTPS on intranet apps, code signing, BitLocker recovery agents, and S/MIME.

In plain English — for every stakeholder

A certificate is a tamper-proof ID card, and a certification authority is the passport office that issues it. You trust a stranger's passport not because you know them, but because you trust the office that stamped it — and checking a certificate is exactly that: follow the stamps up the chain until you reach an office you already trust. Everything in this module is passport-office logistics: who may issue, how cards are handed out, and how stolen ones get canceled.

Validation walks the chain: the presented certificate's signature is checked against its issuing CA's certificate, that one against its issuer, up to a root CA — self-signed, trusted only because it sits in the validator's Trusted Root Certification Authorities store. Alongside signature checks, the validator confirms validity dates, key usage, name constraints, and revocation status (5.5). Break any link — an expired intermediate, an unreachable CRL — and the whole chain fails.

Tier 0, always

AD CS is identity infrastructure. A compromised enterprise CA can mint authentication certificates for any principal — the ESC-series attack techniques against misconfigured templates and CA settings are among the most common AD escalation paths audited today. Treat every CA, and every template's ACL, with domain-controller-grade seriousness.

5.2 · CA types and the two-tier hierarchy

AD CS CAs vary on two independent axes. Enterprise vs standalone: an enterprise CA is AD-integrated — it uses certificate templates, authenticates requesters against the directory, publishes to AD, and supports autoenrollment; it must stay domain-joined and online. A standalone CA knows nothing of AD: no templates, requests contain all their own detail and typically await manual approval — which is precisely what you want for a CA that will spend its life powered off. Root vs subordinate: a root CA signs its own certificate and anchors trust; a subordinate CA is certified by a parent and does the day-to-day issuing.

The standard enterprise design combines them into a two-tier hierarchy:

Why offline? The root's private key is the crown jewel: if it leaks, every certificate in the org is forgeable and recovery means rebuilding trust from zero. A root that is a workgroup machine, powered off in a safe (or a VM on encrypted, offline storage, or better, keys in an HSM), presents no attack surface. Single-tier (one enterprise root doing everything) is defensible only for small labs; three tiers add a policy CA layer that few organizations genuinely need.

In plain English — for every stakeholder

Why keep the root offline? The root CA's private key is the master stamp. If it's ever copied, every ID card the company has ever issued becomes forgeable and you rebuild trust from zero. So the master stamp lives in a safe, powered off, and is brought out once a year for one signature — while an everyday branch office (the issuing CA) handles the daily card printing. Lose the branch office and the root simply disowns it; the company survives.

5.3 · Deploying AD CS

AD CS is a role with several services beyond the CA itself: Web Enrollment (legacy browser enrollment pages), Online Responder (OCSP, 5.5), NDES (SCEP enrollment for network devices and MDM), and CES/CEP (certificate enrollment web services for non-domain or cross-forest clients). Install only what you'll use.

# --- Offline root (standalone, workgroup server) ---
Install-WindowsFeature ADCS-Cert-Authority -IncludeManagementTools
Install-AdcsCertificationAuthority -CAType StandaloneRootCA `
  -CACommonName "Contoso Root CA" -KeyLength 4096 -HashAlgorithmName SHA256 `
  -ValidityPeriod Years -ValidityPeriodUnits 20

# --- Enterprise issuing CA (domain-joined) ---
Install-WindowsFeature ADCS-Cert-Authority, ADCS-Web-Enrollment -IncludeManagementTools
Install-AdcsCertificationAuthority -CAType EnterpriseSubordinateCA `
  -CACommonName "Contoso Issuing CA 01" -KeyLength 4096 -HashAlgorithmName SHA256
# → generates a request file; take it to the root, sign, install the issued cert,
#   then publish the root cert into AD so every domain member trusts it:
certutil -dspublish -f ContosoRootCA.crt RootCA

Pre-deployment decisions that are painful to change later: the CA common name (baked into every issued cert), key length and hash (4096/SHA-256 today), validity periods (root 20y → issuing 5–10y → leaf 1–2y; a CA can't issue certs outlasting its own), and the CDP/AIA URLs — set these to a stable HTTP host you'll keep alive for decades, before issuing anything (5.5).

5.4 · Templates, enrollment, and issuing

On an enterprise CA, every certificate is stamped from a certificate template — an AD-stored blueprint (configuration partition, so forest-wide) defining validity, key size and usage, subject-name construction, and — critically — who may enroll, via the template ACL's Enroll and Autoenroll permissions. You never edit the built-in templates; you duplicate one (e.g., Workstation Authentication → "Contoso Computer Auth v1"), adjust, set permissions, then publish it on the CA (certsrv.msc → Certificate Templates → New → Certificate Template to Issue).

Enrollment paths, in order of how often you'll actually use them:

  1. Autoenrollment — the workhorse. A GPO (Computer and/or User Configuration → Policies → Windows Settings → Security Settings → Public Key Policies → Certificate Services Client – Auto-Enrollment) set to enroll and renew automatically, combined with template Autoenroll permission, silently issues and renews certs for thousands of machines. This is how every DC gets its Kerberos/LDAPS certificate and every laptop gets 802.1X.
  2. MMC enrollment — certlm.msc/certmgr.msc → Request New Certificate, for one-offs.
  3. Request files — certreq -new / -submit / -accept with an INF, for appliances and web servers needing custom SANs.
  4. NDES/SCEP and CES/CEP — devices, MDM-managed mobiles, and off-domain clients.
Template misconfiguration = domain compromise

The single most dangerous checkbox in AD CS is "Supply in the request" for the subject name on a template that also allows client authentication and broad Enroll rights (attack pattern ESC1): any enrollee can request a cert as the domain admin and log on with it. Audit templates for this combination, restrict Enroll permissions, require manager approval where subjects are requester-supplied, and enable CA auditing (certutil -setreg CA\AuditFilter 127).

5.5 · Revocation: CRLs, delta CRLs, and OCSP

Issued certificates sometimes must die early — a laptop is stolen, an employee leaves, a key is exposed. The CA administrator revokes the certificate (with a reason code; Certificate Hold is the only reversible one), and the serial number is published for validators to check via two mechanisms:

Revocation checking mechanisms
MechanismHow it worksTrade-off
CRLThe CA signs and publishes a full list of revoked serials on a schedule (e.g., weekly); clients download from the CDP URL and cache until it expiresSimple and offline-cacheable, but grows large and is only as fresh as its publication interval
Delta CRLFrequent small "changes since the last base CRL" list (e.g., daily) layered on the baseFreshness without full re-downloads; clients need both files
OCSPClient asks the Online Responder about one serial number; signed good/revoked/unknown answerReal-time and tiny, but the responder is now infrastructure that must be highly available
In plain English — for every stakeholder

Revocation is the list of stolen badges taped up at every door. A CRL is that printed list, refreshed on a schedule; a delta CRL is the sticky note of "new since yesterday"; OCSP is phoning the badge office to ask about one badge right now. And note the failure mode in the warning below: if the guard's list expires, he doesn't wave everyone through — he lets no one in. An out-of-date stolen-badge list stops the whole building.

Every issued certificate carries pointers baked in at issuance: the CDP (CRL Distribution Point — where to fetch the CRL) and AIA (Authority Information Access — where to fetch the issuer's cert, and the OCSP URL). This is why 5.3 insisted on configuring stable HTTP URLs first: certificates in the field cannot be updated. Prefer plain HTTP (not HTTPS — chicken-and-egg) on a highly available host, plus LDAP publication for domain members.

The classic AD CS outage

An expired CRL fails validation for every certificate the CA ever issued — 802.1X drops the whole fleet off Wi-Fi, smart card logons fail, LDAPS breaks. Monitor CRL expiry like a certificate expiry, keep overlap periods generous, and remember the offline root must be booted periodically (typically annually) just to re-sign its CRL: certutil -crl, then copy to the CDP.

5.6 · Establishing certificate trust — and protecting keys

Trust is a distribution problem. Inside the forest, an enterprise CA's chain is largely self-distributing: certutil -dspublish placed the root in the directory, from which domain members auto-populate their Trusted Root store, and the enterprise CA registers in the NTAuth store (NTAuthCertificates in the configuration partition) — the special list of CAs whose certificates are allowed to authenticate to AD (smart card / PKINIT logon). Belt-and-suspenders (and for non-domain machines): distribute roots by GPO under Public Key Policies → Trusted Root Certification Authorities.

Trust beyond your forest has escalating options: simply importing a partner's root (trusts everything they issue), a Certificate Trust List or cross-certification with name/policy constraints (trust only certain namespaces or usages), or qualified subordination. The rule: never trust a foreign root more broadly than the business need.

Key archival and recovery

For encryption certificates (EFS, S/MIME), losing the private key means losing the data — so enterprise CAs support key archival: enable a Key Recovery Agent (KRA) certificate on the CA and mark templates to archive the subject's private key at issuance. Recovery deliberately takes two roles (a CA officer retrieves the encrypted blob with certutil -getkey; the KRA decrypts it with certutil -recoverkey) so no single person can silently read everyone's mail. Never enable archival on signing templates — an archived signing key destroys non-repudiation.

Finally, the CA itself: back up regularly (Backup-CARoleService -Path D:\CABackup -Password $pw captures the private key and database — guard that backup like the key it contains), keep issuing-CA keys in an HSM where budget allows, and calendar the CA certificate's own renewal years ahead: as it nears expiry it silently shortens every certificate it issues, which is a strange helpdesk week you can avoid.

Knowledge check · Module 5

Prove certificate services

1. Why is the recommended root CA a standalone CA rather than an enterprise CA?

The whole point of the offline root is zero attack surface. Enterprise CAs are AD-integrated and must remain online domain members; standalone CAs have no such dependency and only need to boot to certify a sub-CA or re-sign the annual CRL.

2. One Monday, every laptop drops off 802.1X Wi-Fi and smart card logons fail — but no certificates were revoked or expired over the weekend. First thing to check?

Fleet-wide simultaneous validation failure is the signature of an expired/unfetchable CRL: revocation checking fails, so every cert from that CA fails validation at once. Re-sign and republish the CRL, then fix the monitoring gap.

3. A template allows requesters to supply their own subject name, grants Enroll to Domain Users, and includes Client Authentication EKU. The risk is…

Requester-supplied subjects + client-auth EKU + broad enrollment is the classic ESC1 template misconfiguration: request a cert with a Domain Admin's UPN, then use PKINIT to become them. Lock the subject to AD-built names, restrict Enroll, or require approval.

4. Key archival on the issuing CA should be enabled for…

Archival exists so encrypted data survives a lost key. Archiving a signing key destroys non-repudiation — a recovered copy means signatures no longer prove the subject made them. Recovery is deliberately split across two roles (CA officer + KRA).
